Validate the containerized unit-test target.

The test stage is a thin Go-toolchain image whose default CMD runs the unit
tests. Build it once, then run it:

docker build --target test -f E:\DockerDesktop\portainer-mcp-safe\docker\Dockerfile -t portainer-mcp-safe-test E:\DockerDesktop\portainer-mcp-safe
docker run --rm portainer-mcp-safe-test

To run a narrower subset, override the CMD, for example:

docker run --rm portainer-mcp-safe-test /usr/local/go/bin/go test -count=1 ./internal/tooldef -run TestGatewayMetadataMatchesEmbeddedTools

Build the local safe-fork runtime image:

This repo is an altered-source fork of the upstream Portainer MCP project:
https://github.com/portainer/portainer-mcp

The upstream license text is retained in the repo root `LICENSE`, and fork
attribution is captured in `NOTICE`.

docker build -f E:\DockerDesktop\portainer-mcp-safe\docker\Dockerfile -t portainer-mcp-safe:0.7.0-safe.1 E:\DockerDesktop\portainer-mcp-safe

Add the server YAML to an MCP Toolkit profile:

docker mcp profile create --name portainer_local
docker mcp profile server add portainer_local --server file://E:\DockerDesktop\portainer-mcp-safe\docker\portainer-mcp-gateway.yaml

Reuse the existing Portainer token secret:

docker mcp secret set portainer-mcp.api_token=<your-portainer-admin-api-token>

Run the gateway with the profile:

docker mcp gateway run --profile portainer_local

This server entry is intentionally pinned to:

- Portainer URL: https://host.docker.internal:9443
- Version check: disabled
- Server identity: portainer-mcp-safe

Why: the local Portainer instance responds as version 2.39.1, while upstream
portainer-mcp v0.7.0 only officially supports 2.31.2, so version validation must
be bypassed. Also, in this Docker Desktop build the gateway did not inject templated
config values from custom file-based server entries, so the working path is to keep
the known-good URL in the YAML.

Gateway-visible config defaults in this fork:

- safe_mode: true
- allow_unredacted_stack_content: false
- allow_sensitive_proxy_paths: false
- proxy_allowlist: empty
- extra_redaction_patterns: empty

Do not keep both `portainer-mcp` and `portainer-mcp-safe` active in the same
profile when reconnecting Cursor. They expose the same tool names, and the
Cursor bridge caches tool descriptors per tool name rather than per server.
Remove the legacy `portainer-mcp` entry first for a clean cutover.

If your Portainer URL changes, edit both allowHosts and the --server-url value in
portainer-mcp-gateway.yaml, then re-add the server to the profile/catalog.

If you change `internal/tooldef/tools.yaml`, run `make regen-gateway-tools` before
rebuilding or publishing so the embedded `tools:` block in
`docker/portainer-mcp-gateway.yaml` stays in sync.

If `make` is not available in your shell, use:

go run ./cmd/regen-gateway-tools

To verify that the gateway YAML is in sync without writing any files:

go run ./cmd/regen-gateway-tools --check

Add --diff to see a line-level diff of any drift:

go run ./cmd/regen-gateway-tools --check --diff

The `tools:` block in `docker/portainer-mcp-gateway.yaml` is auto-generated
(it carries an `# AUTO-GENERATED` marker comment). Do not hand-edit it.
Edit `internal/tooldef/tools.yaml` instead, then rerun the regen tool.

To publish the image to your own registry for a custom catalog:

docker tag portainer-mcp-safe:0.7.0-safe.1 <your-registry>/mcp/portainer-mcp-safe:0.7.0-safe.1
docker push <your-registry>/mcp/portainer-mcp-safe:0.7.0-safe.1

After pushing, update the image field in portainer-mcp-gateway.yaml to that registry reference.

Create and push a custom catalog:

docker mcp catalog create <your-registry>/mcp/portainer-catalog:latest --title "Portainer MCP Safe"
docker mcp catalog server add <your-registry>/mcp/portainer-catalog:latest --server file://E:\DockerDesktop\portainer-mcp-safe\docker\portainer-mcp-gateway.yaml
docker mcp catalog push <your-registry>/mcp/portainer-catalog:latest

Import the catalog in Docker Desktop MCP Toolkit or pull it with:

docker mcp catalog pull <your-registry>/mcp/portainer-catalog:latest

Add Portainer MCP Safe to your existing local custom catalog:

docker mcp catalog server add my-custom-mcp-servers:latest --server file://E:\DockerDesktop\portainer-mcp-safe\docker\portainer-mcp-gateway.yaml

Validation notes from this implementation:

- The image builds successfully from the unified repo root.
- Portainer is reachable from containers at https://host.docker.internal:9443/api/status.
- The MCP profile and custom catalog accept the server YAML, config surface, and token secret wiring.
- Docker MCP Toolkit did not interpolate config templates for this custom local server entry, so the YAML uses a fixed known-good local Portainer URL instead.
- Full tool initialization still requires a real Portainer administrator API token; with a placeholder token the gateway shows "failed to connect: calling initialize: EOF".
